TWEB 2018 - Sérieux
Petite précision concernant les deux méthodes présentées pour github OAuth (Pour ceux qui auraient des soucis avec l’authentification):

La méthode (1): https://heig-vd-tweb.github.io/Teaching-TWEB-2018-Main/slides/2018/05-wrap-up-faq#9
À utiliser si le client et le server se trouvent sur le même domaine (`same-origin`). Car il est difficile de passer des cookies en mode CORS à cause des restrictions des navigateurs. Pour que ça fonctionne en mode CORS on pourrait par exemple utiliser un proxy pour faire croire au navigateur que le client et le server se trouve sur le même domaine.

La méthode (2): https://heig-vd-tweb.github.io/Teaching-TWEB-2018-Main/slides/2018/05-wrap-up-faq#10
Convient en mode CORS ou same-origin. En effet l’access_token n’est pas envoyé par cookie mais dans le body de la réponse. Avec cette méthode vous pouvez persister l’access_token dans le localStorage et vous devrez l’envoyer à votre serveur manuellement à chaques requête.

TWEB 2018 - Sérieux
Et voici une petite synthèse des commentaires qui vous ont été rendus hier par rapport au projet 1 :
- Remarque sempiternel : commentez un tant soit peu votre code, svp.
- Beaucoup d'applications ne sont pas suffisamment ergonomiques (si on se met à la place d'un utilisateur lambda et que le contexte n'est pas correctement présenté ou que l'interface n'est pas assez explicite, le pauvre bougre sera perdu). De manière générale, pensez à guider l'utilisateur (avec des placeholder dans les champs input, par exemple) et à mettre des retours utilisateurs (erreurs, spinners de chargement, etc.).
Typiquement, on clique sur un bouton, on attend, et on se sait pas trop ce qui se passe… Une autre remarque générale : à quoi correspondent les stats que vous affichez ?
- Les tests étaient généralement bâclés, certainement pour des questions de temps et de priorité. Nous allons tâcher de revenir là-dessus en cours, mais de manière générale nous recommandons de tester les contrôleurs de la DB, l'API (endpoints), et les fonctions qui justifieraient un test par leur complexité.
- Attention à la structure des projets : balancer tous les fichiers dans UN dossier n’est jamais une bonne idée.
- Penser à enlever les console.log() qui polluent et qui peuvent donner des informations sensibles sur votre app.
- Attention au délai du rendu ! Nous avons enlevé des points aux gens qui ont rendu après minuit ou qui ont fait des commits injustifiés par la suite.
- Pas mal d'app plantent si nous sélectionnons un utilisateur qui possède un repo vide ou désactivé. Il s’agit certes d’un test un peu tiré par les cheveux, mais si vous réalisez un projet IRL ce genre de cas sera typiquement une source de problèmes et de stress. Et personne ne souhaite ça.
- Même s’il peut s’avérer relou, le linter reste quand même une manière sûre d’avoir un code propre, d’autant plus que vous débutez en JS pour la plupart. Et non, le coup du « on active ESLint mais on désactive la plupart des règles dans le fichier de config » ne passe malheureusement pas 😉
- Attention à ne pas essayer de communiquer directement avec la DB du côté client (cette manipulation est généralement bloquée, mais dans le cas où vous arriveriez à vos fins cela signifie que n'importe qui pourrait faire N'IMPORTE QUEL REQUÊTE auprès de la DB depuis le navigateur en injectant du JS).
- La différence entre le client et le serveur semble encore floue pour certains. Si tel est votre cas, n’hésitez pas à venir nous en parler.

Des bonus ont été accordés pour l'utilisation de charts/maps, pour des UI particulièrement soignées, une bonne présentation, une particulièrement bonne documentation, l’utilisation d’OAuth, d’Angular/React/Vue.js/etc., de D3, la création de scripts de déploiement, l’utilisation d'une API externe, etc. Pour ces raisons, il est possible que vous ayez obtenu la note maximale, même si le projet n’était pas parfait. Nous pensons aussi que l’implication générale et la quantité de travail fournie justifient ces bonnes notes. Encore une fois, merci d’avoir joué le jeu !

TWEB 2018 - Sérieux
Plusieurs d’entre vous ont essayé d’ajouter eslint dans un projet React basé sur create-react-app. Sachez qu’il n’est pas nécessaire d’installer eslint manuellement. Eslint est déjà inclus avec un minimum de règles et vous aurez des erreurs si vous le rajoutez.

Ce que vous devez faire, c’est “customiser” les “rules” selon vos besoins. Soit dans un fichier .eslintrc ou dans le package.json.

Example:
// package.json
"eslintConfig": {
    "extends": "react-app",
    "rules": {
      "no-console": "error"
    }
  },


Plus d’infos: https://facebook.github.io/create-react-app/docs/setting-up-your-editor#displaying-lint-output-in-the-editor
